# 视频数据收集小助手 — 产品概述（方案 B）

## 产品定位

基于 **微信认证服务号** 的轻量视频数据收集工具：主交互全部在 **与服务号的对话框** 中完成（欢迎语、登记、需求下发、答疑、审核结果与提醒）；**自定义菜单**仅用于 **大视频上传** 等对话框能力不足的补充（通常跳转 H5 直连对象存储）。

目标：在合规前提下，标准化引导用户提交可用视频素材，完成用户信息登记、需求下发、审核与跟进，降低人工运营成本（配合关键词规则 + AI Agent 自动话术）。

## 一句话定义

用户在微信里 **和服务号聊天** 完成参与全流程；需要超大文件时走 **菜单 → H5 上传**；公司提供审核、存储与收益/进度通知闭环。

## 核心目标

| 编号 | 目标 |
|------|------|
| G1 | 高效、合规收集符合业务要求的视频，形成可检索的素材库 |
| G2 | 以对话流降低参与门槛，提升响应速度与用户疑虑解答率 |
| G3 | 审核、编号、状态可查，支撑「收集 → 审核 → 结算/跟进」闭环 |
| G4 | 自动话术主导，人工仅处理审核与例外咨询 |

## 交互原则（方案 B 约束）

1. **主线不在菜单、不在推文**：用户路径默认是「发消息 / 收消息 / 传素材」。
2. **菜单极简化**：例如仅「大视频上传」→ H5；其他能力不拆成多层菜单树。
3. **大小视频策略**：对话框可传的范围内直接传；超出限制则 **同一套编号与备注规则** 在 H5 中完成。

## 非目标（V1 不做）

- 独立 APP、复杂小程序游戏化运营
- 以公众号图文运营为主力的互动（推文仅可作引流/公告，不作流程主干）
- 端到端自动内容安全终审（平台规则 + 人工审核仍为底线）
